# Контракты событий

Контракт события - это декларативное описание того, какое событие вообще существует в системе, у какой сущности оно допустимо и как его нужно трактовать.

Контракт сам по себе:

- не является runtime-сущностью;
- не исполняет код;
- не хранит пользовательскую реакцию;
- не подписывается на шину;
- не живёт в `Endge.domain` как обычный документ.

Его задача - быть каталогом допустимых событий.

---

## Что хранит контракт

Типичный контракт описывает:

- `entityType` - у какой сущности есть это событие;
- `eventName` - каноническое имя события;
- `title` - понятное имя для UI;
- `eventKind` - тип события (`lifecycle`, `interaction`, `data` и т.д.);
- `scope` - `owner`, `target` или `global`;
- `supportsBinding` - можно ли к событию привязать `binding`;
- `supportsEnvironmentOverride` - можно ли делать override по окружению;
- при необходимости `payloadSchema` и `contextSchema`.

---

## Чем контракт отличается от события

Контракт и событие - не одно и то же.

### Контракт

Это описание возможности:

- у `Page` бывает `mounted`;
- у `Project` бывает `init`;
- у `View` бывает `ready`.

### Событие

Это уже реальный факт runtime:

- страница `Page#42` смонтировалась;
- проект `Project#1` инициализировался;
- view реально завершил подготовку.

То есть:

- контракт = описание допустимого события;
- runtime event = конкретный произошедший факт.

---

## Чем контракт отличается от binding

`Binding` - это не описание события, а правило реакции.

Например:

- контракт говорит: у `Page` есть событие `mounted`;
- binding говорит: когда это событие случится, нужно выполнить action `load-profile`.

Именно поэтому:

- контракт отвечает на вопрос "что вообще может произойти?";
- binding отвечает на вопрос "что делать, если это произошло?".

---

## Scope: owner и target

`scope` нужен не для того, чтобы сказать, кто запустил процесс, а для того, чтобы понять, у кого именно произошло событие и как оно интерпретируется.

### `owner`

Событие относится к самой сущности.

Примеры:

- `project.init`
- `page.mounted`
- `component.name-change`

Читается так:

- я сам вошёл в это состояние

### `target`

Сущность владеет правилом, но событие происходит у другой сущности.

Пример:

- `Project` реагирует на `page.mounted` дочерней страницы.

Читается так:

- я хочу отреагировать, когда мой target сделал это

---

## Один факт - несколько интерпретаций

Важно: `scope` не создаёт два разных факта мира.

Если смонтировалась `Page#42`, то это:

- один runtime event;
- но он может матчиться как `owner`-событие для самой `Page`;
- и как `target`-событие для `Project`, который хочет реагировать на mounted дочерней страницы.

То есть:

- runtime event один;
- контрактов и реакций может быть несколько.

---

## Роль контрактов в UI

Контракты особенно важны для конфигуратора.

Они позволяют:

- показывать только допустимые события для сущности;
- делить события по смыслу;
- управлять `environment override`;
- не давать пользователю навешивать реакцию на несуществующее событие.

---

## Роль контрактов в целевой архитектуре

В зрелой архитектуре контракт должен быть каноническим описанием runtime-события.

То есть runtime host не должен испускать просто строку, а должен испускать событие, которое:

- существует в contract registry;
- типизировано;
- понятно в UI;
- может быть связано с bindings и actions.

Именно поэтому контракты - это не декоративный слой, а фундамент для:

- UI-конфигурации;
- runtime-dispatch;
- валидации;
- observability;
- общего языка событий в системе.
