# Headless Runtime и Optional Presentation

Эта заметка фиксирует одну из ключевых архитектурных идей: runtime-сущность должна уметь жить и работать даже без визуализации, а отображение должно быть отдельным, опциональным фасетом.

---

## Короткая формула

- `Behavior facet` отвечает за жизнь сущности.
- `Presentation facet` отвечает за то, как сущность отображается.
- Отсутствие presentation не должно ломать behavior.

Иными словами:

- сущность может быть **headless**;
- визуализация подключается отдельно;
- оба фасета живут рядом, но не сливаются в один объект.

---

## Что такое headless runtime

Headless runtime - это runtime-host, который:

- создаётся;
- живёт внутри `Endge.runtime`;
- держит lifecycle;
- публикует события;
- реагирует на данные и bindings;
- не обязан ничего рисовать в UI.

Примеры:

- `Page` может существовать как runtime страницы;
- `Project` может существовать как runtime верхнего уровня;
- `Filter` может жить как логическая сущность без собственного Vue-рендера.

Это важно, потому что поведение и жизненный цикл - не UI-задача.

---

## Что такое optional presentation

Optional presentation - это идея, что визуализация сущности подключается только тогда, когда она действительно нужна.

То есть:

- у сущности может быть runtime без Vue-компонента;
- позже к ней можно прикрепить renderer;
- пока renderer не прикреплён, сущность остаётся рабочей, но headless.

Это даёт гибкость:

- одну и ту же сущность можно использовать в фоне;
- можно подключать разные визуализации для разных проектов и страниц;
- визуальный слой не начинает управлять жизненным циклом сущности.

---

## Почему это полезно

Такое разделение решает сразу несколько задач:

- не смешивает lifecycle с DOM;
- позволяет переиспользовать runtime в разных UI-контекстах;
- даёт возможность переопределять визуал отдельно от поведения;
- делает систему ближе к plugin-friendly модели;
- позволяет сначала поднять runtime, а потом выбрать presentation.

По сути это модель:

`Entity -> Runtime facet -> Optional Presentation facet`

---

## Как это читать для Vue

Удобная Vue-модель здесь такая:

- компонент `EndgePage` поднимает и держит runtime страницы;
- пока компонент смонтирован, runtime живёт;
- component слушает события runtime и пробрасывает их во Vue;
- если для страницы найден presentation renderer, он рендерится;
- если нет, компонент остаётся headless-host оболочкой.

Тогда `EndgePage` - это не просто визуальный компонент, а **runtime shell** для page-сущности.

То же самое может появиться и для других сущностей:

- `EndgeProject`
- `EndgeView`
- `EndgeFilter`
- `EndgeComponentHost`

---

## Что нельзя смешивать

Эта модель становится хрупкой, если смешать в одном месте:

- создание runtime;
- хранение runtime state;
- выбор renderer;
- конкретную Vue-реализацию отображения.

Лучше держать это отдельными слоями:

1. `Runtime host`
2. `Presentation resolver`
3. `Renderer registry`
4. `Vue shell component`

---

## Практическое правило

Если без UI сущность всё ещё имеет смысл и может жить, значит ей нужен headless runtime.

Если UI для неё может меняться без изменения её логики, значит presentation должен быть отдельным фасетом.
