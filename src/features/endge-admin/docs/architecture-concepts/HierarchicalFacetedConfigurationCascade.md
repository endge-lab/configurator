# Hierarchical Faceted Configuration Cascade

Это рабочее название для архитектурной модели, в которой несколько уровней контекста переопределяют друг друга, а у каждого уровня есть несколько фасетов.

---

## Короткая формула

- `Hierarchy` - есть уровни контекста.
- `Faceted` - у каждого контекста несколько аспектов.
- `Configuration` - каждый уровень несёт конфигурацию.
- `Cascade` - итог собирается сверху вниз через override.

Итоговая модель:

`Tenant Context -> Project Context -> Page Context`

Но это не просто три разные сущности предметной области, а три уровня одной и той же логики каскадного контекста.

---

## Что такое context

`Configuration Context` - это слой конфигурации, который:

- хранит собственные настройки;
- имеет родителя в иерархии;
- задаёт значение для своих фасетов;
- может переопределять значения нижестоящих слоёв.

Именно поэтому:

- `tenant`
- `project`
- `page`

можно читать как разные роли одного общего понятия `Configuration Context`.

---

## Что такое facets

`Facet` - это грань одного и того же контекста.

В текущей архитектурной модели минимально есть два фасета:

- `Presentation Facet`
- `Behavior Facet`

### Presentation Facet

Отвечает за:

- шаблоны;
- renderers;
- layout;
- UI-композицию;
- визуальные роли сущности.

### Behavior Facet

Отвечает за:

- lifecycle;
- contracts;
- bindings;
- реакции;
- правила исполнения.

---

## Что такое cascade

`Override Cascade` - это механизм, по которому итоговая конфигурация собирается из нескольких контекстов.

Например:

1. tenant задаёт базовое оформление и общие правила;
2. project уточняет их под конкретное приложение;
3. page доопределяет самый локальный слой.

Итог - это не одна запись, а вычисленный результат каскада.

Для этого полезно различать:

- `Specificity Order` - порядок приоритета слоёв;
- `Effective Configuration` - итоговое состояние после применения всех override.

---

## Почему это не просто inheritance

Это не совсем классическое наследование.

Важнее читать модель как:

- иерархию контекстов;
- набор фасетов;
- вычисление итоговой конфигурации.

То есть нижестоящий слой не просто "наследует" верхний, а получает **effective state**, собранный через cascade.

---

## Что это даёт архитектуре

Такой подход позволяет:

- отдельно обсуждать UI и behavior;
- строить универсальный механизм override;
- переиспользовать одну и ту же модель для разных сущностей;
- не делать `project` и `tenant` специальными исключениями;
- думать о системе как о дереве контекстов, а не как о наборе несвязанных документов.

---

## Каноническая терминология

Ниже набор терминов, которые можно использовать как базовый словарь архитектуры:

- `Configuration Context` - единица контекста и override.
- `Context Hierarchy` - дерево или цепочка контекстов.
- `Presentation Facet` - визуальная грань контекста.
- `Behavior Facet` - поведенческая грань контекста.
- `Override Cascade` - каскад переопределений.
- `Specificity Order` - порядок специфичности слоёв.
- `Effective Configuration` - итоговая собранная конфигурация.
- `Hierarchical Faceted Configuration Cascade` - имя всей модели целиком.

---

## Практическое чтение модели

Если кратко:

- `tenant` - верхний контекст;
- `project` - прикладной контекст;
- `page` - локальный контекст;
- у каждого есть `presentation` и `behavior`;
- итог вычисляется через cascade;
- более специфичный слой уточняет менее специфичный.
