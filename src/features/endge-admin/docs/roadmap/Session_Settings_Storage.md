# Хранение настроек в привязке к сессии пользователя

Потребность: переход от «все настраиваемые данные в localStorage на устройстве» к версионированному хранилищу, привязанному к сессии пользователя, с возможностью очистки по пользователю и по сессии и автоматическим сбросом при повреждении данных. Не путать с конфигурацией среды и feature flags — см. **Configuration_And_Feature_Flags** и **Variables_Env_Override**.

## Обоснование

В мультипользовательских и enterprise-системах персональные и сессионные настройки (раскладка панелей, открытые вкладки, выбранная локаль) не должны смешиваться между пользователями на одном устройстве и должны поддаваться централизованной очистке (GDPR, смена владельца устройства, инциденты). Отраслевая практика: хранение таких настроек по userId/sessionId на бэкенде или в ключах локального хранилища с префиксом; версионирование формата и безопасный парсинг предотвращают падение приложения из-за повреждённых данных. Риск при отсутствии: утечка контекста между пользователями, невозможность «выйти и забыть» или «очистить всех пользователей» для администратора.

## Текущее состояние

Сейчас все настраиваемые и сериализуемые данные сохраняются в **localStorage** без привязки к пользователю и сессии:

- **endge-app** — выбранный проект и среда (EndgeApp).
- **endge:parameters** — хранилище параметров/фильтров (FiltersStore) в админке и в приложении.
- **endge:filters** — данные фильтров (Filter_Inspector, FilterGenerator).
- **endge-editor-tabs**, **app:grid-layout-state** — вкладки редактора и состояние grid layout админки.
- **endge:parameters[identity]** — значения параметров по identity (endge-domain, инспекторы).
- Состояние домена (endge-domain.ts) — снимки и подобное.
- **replaceme:locale** — выбранная локаль (i18n).
- Настройки колонок RevoGrid (localStorage по ключам runtimeId-index), состояние сортировки таблиц.
- **endge:isLocalTime**, **app:date-format** — локальное время, формат даты.
- **zoom** (Endge.ui) — масштаб.
- **notification_last_read**, **endge:theme** — уведомления, тема.
- Состояние Smart Tabs (ключ вида endge-editor-tabs) — уже с версией (v: 1); при несовпадении версии данные отбрасываются.

Итог: один и тот же браузер/устройство разделяют все эти ключи между всеми пользователями; нет возможности «очистить настройки для пользователя X» или «сбросить только эту сессию»; при повреждении JSON часть ключей может сломать загрузку, и не везде есть защита через версию и безопасный парсинг.

## Требования (To Do)

### 1. Привязка к сессии пользователя

Все настройки, которые сейчас сериализуются в localStorage, должны храниться **в привязке к сессии пользователя**:

- Идентификатор пользователя (userId из auth) и идентификатор сессии (sessionId) должны быть частью ключа или пространства имён при сохранении/загрузке.
- Один пользователь с нескольких устройств/браузеров может иметь несколько «сессий»; каждая сессия имеет свой набор сохранённых настроек.
- При логине без сессии (или при анонимном режиме, если он есть) можно использовать временный sessionId или fallback на устройство (текущее поведение), но предпочтительно явно разделять «настройки устройства» и «настройки сессии пользователя».

Технически это может быть:

- **Вариант A**: бэкенд — настройки хранятся на сервере по userId/sessionId; фронт при загрузке запрашивает их, при изменении — отправляет. Полный контроль очистки на стороне сервера.
- **Вариант B**: фронт — ключи localStorage/sessionStorage включают userId и sessionId (например, `endge-app:${userId}:${sessionId}`), так что данные разных пользователей и сессий не смешиваются. Очистка «всех пользователей» или «конкретной сессии» тогда делается либо через бэкенд (если храним копию на сервере), либо через админский инструмент, который знает список userId/sessionId и удаляет соответствующие ключи (если они известны и хранятся локально).
- **Вариант C**: гибрид — часть настроек только на фронте с префиксом по userId/sessionId, часть — на бэкенде с привязкой к сессии; при первом заходе после логина подтягивать с сервера и мержить с локальными.

Для enterprise (в т.ч. «очистка всех пользователей») предпочтительна серверная или гибридная модель, чтобы администратор мог очищать данные централизованно.

### 2. Очистка всех пользователей и очистка конкретной сессии

- **Очистка всех пользователей**: возможность (через админку или API) удалить все сохранённые настройки по всем пользователям. Если хранилище на бэкенде — удаление по всем userId/sessionId; если локальное — удаление всех ключей, входящих в зарегистрированный список префиксов/паттернов платформы.
- **Очистка конкретной сессии пользователя**: возможность удалить настройки только для одной выбранной сессии (например, «выйти и забыть всё с этого устройства» или «сбросить сессию пользователя X»). Требует хранения маппинга «сессия → ключи» или единого пространства ключей по sessionId и удаления по нему.

В To Do платформы должно быть явно: реализовать API/действия «очистить настройки по всем пользователям» и «очистить настройки по sessionId (и опционально по userId)».

### 3. Версионирование и автоматический сброс при повреждении

- **Версионирование**: каждая сохраняемая структура должна иметь версию (схемы/формата), как уже сделано для Smart Tabs (`v: number`). При изменении формата данных версия увеличивается; при загрузке, если версия не совпадает или отсутствует — считать данные устаревшими и не применять их (или мигрировать, если предусмотрена миграция).
- **Защита от сломанных данных**: при чтении из хранилища (localStorage или ответ сервера) всегда использовать безопасный парсинг (try/catch, валидация типа). Если парсинг не удался или данные не проходят валидацию:
  - не применять их;
  - очистить этот ключ/запись (чтобы не падать при следующем чтении);
  - при необходимости отправить событие в диагностику (канал `storage`, тип «corrupted»);
  - дать пользователю начать «с чистого листа» — дефолтные настройки, без восстановления старого состояния.
- **Единый контракт**: все места, где сейчас пишут в localStorage напрямую, должны проходить через общий слой «session settings storage» (или «user session preferences»), который:
  - добавляет к ключу префикс/namespace по userId и sessionId (или обращается к API бэкенда);
  - добавляет версию при записи;
  - при чтении проверяет версию и целостность и при ошибке очищает запись и возвращает дефолт.

Итог: настройки версионированы; при сломанной или несовместимой версии они автоматически очищаются и пользователь заново настраивает окружение без ручного очищения localStorage.

### 4. Каталог сериализуемых настроек

В To Do целесообразно вести **каталог всех типов настроек**, которые сериализуются (сейчас в localStorage), чтобы:

- ни один ключ не остался без миграции на «session-scoped» и версионирование;
- было ясно, какие ключи участвуют в «очистке всех пользователей» и «очистке сессии».

Пример (по текущему коду): endge-app, endge:parameters, endge:filters, endge-editor-tabs, app:grid-layout-state, replaceme:locale, ключи RevoGrid, endge:isLocalTime, app:date-format, zoom, notification_last_read, endge:theme, состояние домена и т.д. Каждый пункт каталога — кандидат на перенос в единый слой хранения с привязкой к сессии и версией.

## Риски и ограничения

- Миграция существующих ключей localStorage на формат с userId/sessionId и версией потребует однократной логики переноса или признания старых ключей невалидными (пользователь заново настраивает). При серверном хранении — зависимость от API при старте и при каждом изменении настроек; нужен fallback при недоступности сети.
