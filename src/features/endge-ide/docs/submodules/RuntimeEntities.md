# Runtime-сущности

Runtime-сущность - это живой объект исполнения, который создаётся на основе доменной конфигурации и существует внутри runtime-графа приложения.

Если упростить:

- доменная сущность отвечает за описание;
- runtime-сущность отвечает за реальную работу.

---

## Слои модели

Чтобы не путаться, полезно разделять четыре слоя.

### 1. Reflect / domain entity

Это документ или reflect-модель:

- `Project`
- `Page`
- `View`
- `Query`
- `Component`

Тут хранится конфигурация, связи и метаданные.

### 2. EventContract

Это описание, какие события допустимы для сущности.

### 3. Binding

Это правило реакции на событие.

### 4. Runtime entity

Это живой host, который:

- запускается;
- слушает изменения;
- испускает события;
- уничтожается;
- связан с `RaphNode`.

---

## Что есть сейчас

В текущем ядре наиболее явно оформлены runtime-root для:

- `query`
- `table`
- `action`

Они строятся как `RuntimeHost`, работают внутри `Raph` и имеют собственную логику исполнения.

---

## Что является runtime-хостом

Runtime-хост обычно имеет:

- ссылку на исходную entity;
- внутреннее состояние исполнения;
- доступ к данным и dependencies;
- lifecycle методы;
- связь с event bus;
- привязку к фазам `Raph`.

То есть runtime-хост - это не данные о сущности, а механизм её жизни внутри работающего приложения.

---

## Почему это важно для контрактов и bindings

Именно runtime-сущность является источником реального факта мира.

Например:

- `Page` в конфигураторе просто описана;
- а runtime-host страницы реально доходит до mounted;
- в этот момент он сообщает в ядро о событии;
- дальше контракт говорит, как это событие понимать;
- binding говорит, что на него делать.

Без runtime-host нельзя получить честный lifecycle. Будет только статическая схема без исполнения.

---

## Runtime entity и RuntimeScope

Это тоже разные вещи.

- `Runtime entity` - живой объект уровня `query / table / action / page / view`.
- `RuntimeScope` - контекст исполнения кода, переменных и вложенных вычислений.

Scope может создаваться внутри runtime entity, но он не заменяет сам runtime host.

---

## Куда идёт архитектура

Целевой путь обычно такой:

- все ключевые сущности получают собственный runtime-host;
- все runtime-host испускают канонические события;
- dispatcher одинаково обрабатывает `owner / target / global`;
- `Endge.events` остаётся transport-слоем для кода;
- пользователь настраивает реакции не в коде, а через bindings во вкладках редакторов.
