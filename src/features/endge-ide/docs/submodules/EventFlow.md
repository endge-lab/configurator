# Поток событий и реакций

Эта страница объясняет, как связываются между собой runtime-сущности, контракты, bindings, actions и `Endge.events`.

---

## Короткая схема

Общий поток выглядит так:

1. runtime-host переживает реальный lifecycle-факт;
2. ядро оформляет его как каноническое событие;
3. contract registry определяет, какие трактовки события допустимы;
4. bindings resolver собирает подходящие реакции;
5. `Endge.flow` исполняет нужные action/script;
6. событие может быть проброшено в `Endge.events` для подписчиков из кода.

---

## Что является источником

Источником всегда является не контракт и не binding, а runtime-сущность.

Например:

- runtime страницы дошёл до mounted;
- runtime запроса получил результат;
- runtime action завершил шаг.

Это и есть первичный факт.

---

## Где участвуют контракты

Контракты включаются на этапе интерпретации.

Они отвечают на вопросы:

- допустимо ли такое событие вообще;
- как оно называется канонически;
- для какой сущности его можно показывать в UI;
- является ли оно `owner`, `target` или `global`.

Контракт не делает side effect. Он только задаёт смысл и рамки.

---

## Где участвуют bindings

После того как событие признано допустимым, система ищет реакции.

`Binding` хранит:

- owner;
- target;
- имя события;
- script / action;
- режим `replace / append / prepend / disable`;
- приоритет;
- environment override.

Resolver собирает итоговый список bindings и применяет правила наследования.

---

## Что выполняется раньше: bindings или шина

В целевом архитектурном чтении сначала срабатывает ядро конфигурируемых реакций, а потом шина:

1. runtime сообщает о событии;
2. ядро dispatch-ит канонический event;
3. `Endge.bindings` исполняет declarative bindings;
4. затем `Endge.events` даёт возможность коду тоже отреагировать на тот же факт.

Это важно, потому что binding - это доменная политика, а event bus - transport для кода и наблюдения.

---

## Где находится Raph

`Raph` находится уровнем ниже этого flow.

Он не решает, какие события допустимы и какие actions надо запускать. Его задача:

- доставить data-change до runtime-host;
- пометить dirty-ноды;
- прогнать фазы;
- вызвать соответствующий runtime.

А уже runtime-host превращает это в смысловой факт, который потом проходит через contracts, bindings и event bus.

---

## Один `page.mounted` и несколько реакций

Если mounted случился у страницы, возможна такая картина:

- страница реагирует на него как `owner`;
- проект реагирует на него как `target`;
- кодовые подписчики получают его через `Endge.events`.

Но это всё ещё один runtime-факт, а не три независимых события.

Отличается только угол интерпретации и набор последующих реакций.

---

## Зачем разделять эти слои

Такое разделение даёт системе сразу несколько преимуществ:

- UI может настраивать реакции без кода;
- runtime остаётся честным источником lifecycle-событий;
- event bus можно использовать отдельно для технических подписчиков;
- contracts дают единый словарь событий;
- bindings можно наследовать, отключать и переопределять по окружению.

---

## Как это должен читать пользователь

Удобная ментальная модель такая:

- `runtime entity` - сущность реально живёт;
- `contract` - мы договорились, как называть этот факт;
- `binding` - мы настроили, что делать на этот факт;
- `action` - выполнилась конкретная реакция;
- `Endge.events` - код тоже может это услышать.
