# Как учить агента внедрять изменения в проект

Ниже — по шагам, от текущего состояния до «агент предлагает действие → пользователь применяет его в один клик» и дальше.

---

## Сейчас: агент только подсказывает

- Пользователь пишет в чат, например: «Сформируй модельную сущность для вида “Список заказов”».
- Модель отвечает текстом и/или JSON (например, описание полей, структура).
- **Связь с приложением:** её нет. JSON или код нужно вручную скопировать и вставить в нужный редактор или в Payload.

То есть мы **не привязываем** ответ агента к конкретным функциям приложения — пользователь сам решает, куда это вставить и как вызвать создание сущности.

---

## Шаг 1: Структурированный ответ + одна кнопка «Применить»

Идея: договориться с моделью о формате ответа и на фронте обрабатывать только такие сообщения.

1. **Формат ответа агента**  
   Модель в конце ответа (или в отдельном блоке) отдаёт структурированный JSON, например:
   ```json
   {
     "action": "create_entity",
     "entityType": "view",
     "payload": {
       "id": "orders-list",
       "name": "Список заказов",
       "displayName": "Список заказов",
       "folderId": "..."
     }
   }
   ```
   Или для компонента: `"entityType": "component-table"`, `payload` — в формате, который у вас принимает `DocumentFactory.create` / `fromPlain` для этого типа.

2. **Фронт**  
   - Парсим последнее сообщение ассистента: ищем JSON-блок (например, в markdown-блоке кода с меткой `application/json` или между маркерами ```` ```json ... ``` ````).
   - Если нашли и `action === "create_entity"` (и при необходимости тип сущности из белого списка) — показываем под сообщением кнопку **«Применить»**.
   - По клику:
     - `entityType` и `payload` маппим в вызов существующего API создания документа:
       - `DocumentFactory.create(entityType, { id, name, folderId, ... })` + `Endge.schema.saveDocument(id, entityType, { model: draft })`  
       как в `CreateDocument_Modal.vue`,  
       или
       - для более сложных сущностей — восстановление из plain через `fromPlain` (если есть) и затем сохранение через `Endge.schema.saveDocument` / аналог.
   - После успешного применения — toast «Сущность создана», при желании открыть созданный документ в редакторе (как сейчас делается через `EndgeAdmin.tabs.openDocument`).

Так мы **привязываем** ответ агента к одной конкретной функции: «создать сущность по этому JSON». Пользователь не вставляет JSON руками — нажимает «Применить».

---

## Шаг 2: Не только «создать», но и «обновить» / другие действия

- Вводим другие значения `action`: например `update_entity`, `create_folder`, `add_component_to_page`.
- Для каждого типа действия:
  - описание формата (какие поля в `payload`) пишем в системный промпт агента и/или в документацию для разработчиков;
  - на фронте — отдельный обработчик: по `action` вызываем нужную функцию (та же схема: маппинг `payload` в вызовы `Endge.schema.*`, фабрик, редакторов).
- Под сообщением с валидным JSON снова показываем кнопку «Применить» (или «Применить: создать вид», «Применить: обновить компонент» — по типу действия).

Так агент «знает» только формат ответа; **привязка к функциям** — целиком на стороне фронта (один диспетчер по `action` + существующие API домена).

---

## Шаг 3: Function calling (инструменты) на бэкенде

Более продвинутый вариант: модель не просто печатает JSON в ответе, а вызывает «инструменты» (function calling).

1. **Бэкенд (assistance-service)**  
   В запрос к Ollama (или другой LLM) передаётся список инструментов, например:
   - `create_entity(type, payload)` — создать сущность;
   - `get_entity_schema(type)` — вернуть схему сущности (для подсказок модели).
   Модель возвращает вызовы вида: «вызвать `create_entity` с такими аргументами».

2. **Бэкенд** выполняет вызов (создаёт сущность в своей копии схемы или отдаёт команду фронту).

3. **Фронт** либо:
   - получает от бэкенда уже «применённый» результат (например, «документ создан, id = …»), или
   - получает структурированный ответ «нужно выполнить create_entity с payload …» и сам выполняет его теми же вызовами, что в шаге 1–2.

Привязка к функциям тогда делится: **бэкенд** знает список инструментов и их сигнатуры; **фронт** по-прежнему знает, как отобразить результат и как вызвать `DocumentFactory` / `saveDocument` / открытие вкладки, если применение делается на фронте.

---

## Где что делать в нашем проекте

- **Системный промпт (assistance-service)**  
  Описать формат ответа для «рекомендуемых действий»: когда модель хочет предложить создание/изменение сущности, она должна вернуть JSON с полями `action`, `entityType`, `payload` (и при необходимости `description` для отображения пользователю). Можно добавить примеры для 2–3 типов сущностей (view, component-table, folder и т.д.).

- **Фронт: виджет чата**  
  - Парсинг последнего сообщения ассистента: извлечь JSON из markdown/code block.
  - Валидация: `action` из белого списка, `entityType` — один из известных типов документов, `payload` — объект.
  - Кнопка «Применить» (или «Создать сущность» / «Применить изменение») только при валидном таком блоке.
  - Обработчик клика: маппинг в `DocumentFactory.create` + `Endge.schema.saveDocument` (и при необходимости `fromPlain` для сложных сущностей), затем toast и открытие вкладки.

- **Дальше**  
  - Расширить набор `action` и типов сущностей.  
  - При желании перенести «интерпретацию намерения» на бэкенд через function calling, оставив на фронте только выполнение «применить» по полученной команде.

---

## Кратко

| Вопрос | Ответ |
|--------|--------|
| Сейчас JSON вставляем вручную? | Да. Агент только генерирует текст/JSON, пользователь сам решает, куда вставить и как вызвать создание сущности. |
| Как привязать к функциям? | Договориться о формате ответа (например `action`, `entityType`, `payload`), парсить его во фронте и по `action` вызывать уже существующие функции: `DocumentFactory.create`, `Endge.schema.saveDocument`, при необходимости `fromPlain`. |
| Куда встроить первый шаг? | Виджет чата: парсинг JSON из ответа → кнопка «Применить» → один обработчик «создать сущность» через текущий API создания документов. |

После этого можно плавно добавлять новые типы действий и при необходимости подключать function calling на бэкенде.
